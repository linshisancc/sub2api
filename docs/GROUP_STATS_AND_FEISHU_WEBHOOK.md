## 分组账号用量展示

### 功能说明

管理员与普通用户均可在仪表盘的「分组账号用量」区域，按分组查看各账号的实时用量窗口进度条，与账号管理列表的展示风格完全一致。

| 位置 | 展示范围 | 展示内容 |
|------|---------|---------|
| **管理员看板**（`/admin/dashboard`） | 系统中所有活跃分组及其活跃账号 | 每个账号的 5h / 7d 用量进度条（utilization %） |
| **普通用户看板**（`/dashboard`） | 当前用户可用分组（订阅 + 公开/允许分组）及其活跃账号 | 同上，账号用量字段与管理员视图完全一致（不脱敏） |

> 默认所有分组**全部展开**；账号列表按分组懒加载，组件内缓存避免重复请求。API 密钥管理列表不展示分组消耗数据（一个 API 密钥可关联多个分组，无法对应展示）。

### 界面说明

**管理员看板 — 分组账号用量组件**：

```
┌─────────────────────────────────────────────────┐
│ 分组账号用量                          3 个分组   │
├─────────────────────────────────────────────────┤
│ ▶ anthropic-default  [anthropic]  12            │
├─────────────────────────────────────────────────┤
│ ▼ openai-default     [openai]      5            │
│   ┌──────────────────────────────────────┐      │
│   │ my-account-1  oauth                  │      │
│   │   5h  ████░░░░  42%  2h 30m          │      │
│   │   7d  █░░░░░░░  18%  5d 12h          │      │
│   ├──────────────────────────────────────┤      │
│   │ my-account-2  oauth                  │      │
│   │   5h  ████████  98%  0h 15m          │      │
│   │   7d  ████░░░░  56%  3d 02h          │      │
│   └──────────────────────────────────────┘      │
└─────────────────────────────────────────────────┘
```

- 点击分组行展开 / 收起账号列表，**所有分组默认展开**（管理员与普通用户视图一致）。
- 账号列表懒加载（展开时触发），加载完成后缓存，不重复请求。
- 进度条颜色：绿色（0–79%）→ 琥珀色（80–99%）→ 红色（100%+）。
- 百分比含义：该账号在对应时间窗口内的**用量 / 配额上限**，与账号管理列表语义一致。
- 普通用户视角下，分组徽标数字来自展开后加载的账号数（用户侧分组接口不携带 `active_account_count`）。

### 数据来源

组件按当前用户角色调用不同接口：

**管理员视角**

| 调用 | 用途 |
|------|------|
| `GET /api/v1/admin/groups/all` | 获取全部活跃分组列表 |
| `GET /api/v1/admin/accounts?group=<id>&status=active&page_size=50` | 获取指定分组的活跃账号 |
| `GET /api/v1/admin/accounts/<id>/usage` | 获取单账号用量窗口数据（由 `AccountUsageCell` 内部发起） |

**普通用户视角**

| 调用 | 用途 |
|------|------|
| `GET /api/v1/groups/available` | 获取当前用户可绑定的分组列表（基线与 API Key 创建页一致） |
| `GET /api/v1/groups/<id>/accounts` | 获取该分组下的活跃账号摘要；分组不在可用集合时 403 |
| `GET /api/v1/accounts/<id>/usage` | 获取单账号用量；账号不属于任一可用分组时 403 |

### 前端组件

- **`GroupAccountsWidget.vue`** — `frontend/src/components/dashboard/GroupAccountsWidget.vue`  
  分组账号用量主组件，支持 `mode="admin" | "user"`：admin 走 `adminAPI`，user 走 `userGroupsAPI`，所有分组默认展开。
- **`AccountUsageCell.vue`** — `frontend/src/components/account/AccountUsageCell.vue`  
  账号用量单元格，支持 Anthropic、OpenAI、Gemini、Antigravity 等多平台；可选 `usageFetcher` prop 用于在用户视角下切换到 `/api/v1/accounts/:id/usage`。
- **`UsageProgressBar.vue`** — `frontend/src/components/account/UsageProgressBar.vue`  
  带颜色状态和剩余时间的进度条组件。

### 后端 Handler

- **`UserAccountHandler`** — `backend/internal/handler/user_account_handler.go`  
  用户侧分组账号 / 单账号用量查询。权限基线复用 `APIKeyService.GetAvailableGroups`。

## 飞书 Webhook 告警

### 功能说明

在原有邮件告警的基础上，支持通过飞书机器人 Webhook 推送告警消息。每类告警内置冷却机制，防止同一告警在短时间内重复推送（默认 30 分钟内只推送一次）。

### 支持的告警类型

| 告警类型 | 触发条件 |
|---------|---------|
| **用户余额不足** | 用户余额低于设定阈值时触发（与余额低通知联动） |
| **账号额度超限** | 账号日/周/总额度触达告警阈值时触发（与账号额度通知联动） |
| **账号被限流** | 账号被上游服务商限流（收到 429、5h/7d 用量窗口打满等），进入「限流中」状态时触发 |
| **账号限流恢复** | 账号退出「限流中」状态（窗口重置自动恢复，或管理员手动清除限流）时触发 |
| **监控告警恢复** | 「监控」模块的告警策略指标回落到阈值内、告警从 firing 转为 resolved 时触发 |

> **「账号额度超限」与「账号被限流」的区别**：前者是 sub2api 后台为账号配置的**日/周/总额度上限**（美元金额），随计费累加跨过阈值时触发，仅对 APIKey/Bedrock 账号生效；后者是**上游服务商的原生限流**（如 Anthropic 5h/7d 窗口、OpenAI/Gemini 429），由账号进入「限流中」运行时状态触发，对所有账号类型生效。两者是相互独立的信号。

### 配置步骤

#### 1. 创建飞书机器人 Webhook

1. 打开飞书，进入目标群聊
2. 点击右上角「设置」→「机器人」→「添加机器人」→「自定义机器人」
3. 填写机器人名称（如 `Sub2API 告警`），完成创建
4. 复制生成的 Webhook URL，格式如：
   ```
   https://open.feishu.cn/open-apis/bot/v2/hook/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
   ```

#### 2. 在 Admin 设置页配置

进入管理员后台 → **系统设置** → **飞书 Webhook** Tab（独立 Tab，不在邮件设置内）：

**基础配置卡片**：

| 配置项 | 说明 | 默认值 |
|-------|------|--------|
| 启用飞书 Webhook | 全局开关 | 关闭 |
| Webhook URL | 飞书机器人的 Webhook 地址 | — |
| 冷却时间（分钟） | 同类告警触发后，冷却期内不重复推送 | 30（最大 1440） |
| @所有人 | 推送告警卡片时 @所有人 | 关闭 |
| @指定成员 | 推送时 @ 指定成员，填写飞书 open_id（每行一个，也支持逗号/分号分隔） | — |

**推送告警类型卡片**（启用后显示）：

| 配置项 | 说明 | 默认值 |
|-------|------|--------|
| 用户余额不足 | 是否推送余额不足告警 | 开启 |
| 账号额度超限 | 是否推送账号额度超限告警**及账号被限流告警**（两类共用此开关） | 开启 |
| 监控告警 | 是否推送「监控」模块告警策略命中的告警（详见下文「监控告警推送飞书」） | 关闭 |

> **注意**：
> - 余额不足告警需要先在「邮件设置」中启用余额低通知全局开关；账号额度超限需要在「邮件设置」中启用并配置账号配额通知阈值。
> - 「账号被限流」告警复用「账号额度超限」开关，无需单独配置，也不依赖任何邮件设置；账号一旦进入限流状态即推送。

### 消息格式

推送消息采用飞书**交互式卡片**（`msg_type: interactive`）：带颜色标题栏（触发类红/橙、
恢复类绿色）、emoji 图标、字段加粗排版；若配置了 @所有人 / @指定成员，卡片底部追加
@提醒块。下方示例以纯文本示意卡片内容：

**余额不足告警**（💰 橙色）：
```

用户：alice（alice@example.com）
当前余额：$0.0123
告警阈值：$1.0000
```

**账号额度超限**（📊 橙色）：
```

账号：my-anthropic-account
平台：anthropic
维度：日限额 / Daily
当前用量：$9.8765 / 告警阈值：$9.5000
```

**账号被限流**（⏳ 红色）：
```

账号：my-anthropic-account
平台：anthropic
状态：限流中
预计恢复时间：2026-05-15 18:30:00
```

**账号限流恢复**（✅ 绿色）：
```

账号：my-anthropic-account
平台：anthropic
状态：已恢复
恢复时间：2026-05-15 18:45:00
```

### 冷却机制说明

- 冷却以「告警类型 + 实体 ID」为维度独立计算
  - 余额告警：每个用户独立冷却
  - 账号额度告警：每个账号独立冷却
  - 账号限流告警：每个账号独立冷却（与账号额度告警的冷却键相互独立，互不影响）
  - 账号限流恢复告警：每个账号独立冷却，冷却键与「账号限流告警」相互独立（触发与恢复互不抵消）
- 冷却基于 Redis，重启服务不会重置冷却状态（TTL 保存在 Redis 中）
- 冷却时间可在设置页调整，最小 1 分钟，最大 1440 分钟（24 小时）

### 账号限流恢复触发机制

「账号限流恢复」通知有以下两条触发路径（任一满足即推送）：

| 路径 | 说明 |
|------|------|
| **被动请求触发** | 账号限流期间窗口已重置，下一个请求路由到该账号时，上游返回正常响应，系统检测到限流已解除并清除状态 |
| **后台扫描触发（2 分钟周期）** | 后台扫描器每 2 分钟检查一次所有 `rate_limit_reset_at` 已过期的账号，自动执行清除并推送恢复通知，覆盖「窗口到期后无请求流量」的情况 |

后台扫描器适用于所有平台（Anthropic、OpenAI、Gemini 等），只要账号设置了 `rate_limit_reset_at` 字段，窗口到期后均会被扫到。

> **实现注意**：`ClearRateLimit()` 内部判断"是否需要推送恢复通知"时，必须检查
> DB 中 `rate_limit_reset_at` 是否仍非空，而**不能**使用 `IsRateLimited()`（其语义是
> `time.Now().Before(*resetAt)`，在窗口刚过期时返回 `false`）。否则后台扫描器扫到的
> "窗口已过期"账号会全部被这条门槛拦掉，恢复通知永远不会发出 —— 表象就是：限流触发
> 通知收到了、到点后无新请求、恢复通知缺失。被动请求路径（有流量打到账号时清理）
> 不受影响，因此问题只在"窗口到期后无流量"场景下复现。

#### 扫描频率（如何调整）

后台扫描周期目前**硬编码为 2 分钟**，位置 `backend/internal/service/ratelimit_service.go:1492`：

```go
ticker := time.NewTicker(2 * time.Minute)
```

如需修改，可选三种方式：

| 方式 | 适用场景 | 成本 |
|------|---------|------|
| 直接改常量 | 团队内部敲定一个固定值 | 低（需重新构建镜像） |
| 抽成环境变量（如 `RATE_LIMIT_RECOVERY_SCAN_INTERVAL`），`time.ParseDuration` 读取，默认 2 分钟 | 自托管想留运维旋钮 | 中（改 `.env` + 重启 sub2api 容器即可） |
| 存入 `settings` 表，扫描器按需重建 ticker | 频率需要在 Admin 页热更 | 高 |

取值建议：

- **下限 ≥ 30 秒**：扫描会全表过 `rate_limit_reset_at IS NOT NULL AND <= now()`，过密只会徒增 DB 负载；飞书消息间隔本身已有冷却（默认 30 分钟）兜底，扫得再勤也不会更早通知。
- **上限 ≤ 5 分钟**：恢复通知的延迟约为扫描周期的 0.5×–1× 倍，过长会让用户感觉"窗口都过去 5 分钟了还没收到恢复消息"。
- 当前 2 分钟为推荐默认。

### 与邮件告警的关系

飞书告警与邮件告警**并行触发**，互不影响：

| | 邮件告警 | 飞书 Webhook |
|-|---------|-------------|
| 余额不足 | 发送给用户本人及其额外邮箱 | 推送到飞书群 |
| 账号额度超限 | 发送给管理员通知邮箱 | 推送到飞书群 |
| 账号被限流 | 无邮件告警 | 推送到飞书群 |
| 控制开关 | 各自独立 | 各自独立 |

## 监控告警推送飞书

### 功能说明

监控告警模块（管理员后台「监控」页的告警策略）原本只支持**邮件**通知。现支持在邮件之外，
同时把命中的告警策略推送到飞书群。

> **与上文飞书告警的区别**：上文的余额/账号类告警是业务事件触发；本节是「监控」模块的
> 指标告警策略（成功率、错误率、CPU、限流账号数等）命中阈值时触发，两者来源不同。

### 配置步骤

1. 先在「系统设置 → 飞书 Webhook」中启用飞书 Webhook 并填写 Webhook URL（与上文共用同一地址）。
2. 在同一页的「推送告警类型」卡片中，打开「监控告警」开关（与「用户余额不足」「账号额度超限」并列）。

> 监控告警的飞书推送由「推送告警类型 → 监控告警」开关（配置键 `feishu_webhook_notify_ops`）控制，
> 并复用「监控 → 告警通知配置」中的**静默规则**；**不**受邮件「最低级别（MinSeverity）」约束，
> 也不依赖、不影响邮件收件人配置。监控页自身没有飞书相关开关。

### 行为说明

- 飞书推送与邮件通知**并行触发**，互不影响：关闭其一不影响另一。
- 推送门槛：仅受静默规则覆盖；不受邮件最低级别（MinSeverity）约束 —— 调高邮件最低级别不会屏蔽飞书推送。
- 不复用上文飞书告警的 30 分钟冷却 —— 监控告警模块自身已有沉降/冷却/单活动事件去重机制，
  避免双重门槛误吞告警。
- **无流量自动恢复**：当告警时间窗口内没有请求数据时（如限流结束后流量归零），评估器会将活跃
  的 firing 事件自动置为 resolved 并推送恢复通知，不会因「无数据」而导致告警永久挂起。

### 消息格式

**触发**（🚨 颜色按 severity：P0 红 / P1 橙 / P2 黄 / 其他 灰）：
```
🚨 [Sub2API] 运维告警（P1）
规则：API 成功率过低
指标：success_rate < 95.00（当前 88.30）
状态：firing
触发时间：2026-05-15 18:30:00
说明：success_rate < 95.00 (current 88.30) over last 5m (anthropic)
```

**恢复**（✅ 绿色）：告警指标回落到阈值内、事件转为 resolved 时推送：
```
✅ [Sub2API] 运维告警恢复（P1）
规则：API 成功率过低
指标：success_rate < 95.00
状态：已恢复
触发时间：2026-05-15 18:30:00
恢复时间：2026-05-15 18:42:00
```

> 监控告警恢复推送复用「监控告警」开关（`feishu_webhook_notify_ops`）与监控模块的静默规则，
> 不引入额外冷却（监控模块单活动事件去重已保证不重复）。

## 相关配置键（仅供参考）

飞书 Webhook 设置存储在 `settings` 表中：

| Key | 说明 |
|-----|------|
| `feishu_webhook_enabled` | `true` / `false`，飞书 Webhook 全局开关（业务告警与监控告警共用） |
| `feishu_webhook_url` | Webhook URL 字符串（业务告警与监控告警共用） |
| `feishu_webhook_cooldown_minutes` | 整数，单位分钟（仅作用于上文业务告警，监控告警不使用） |
| `feishu_webhook_notify_balance` | `true` / `false`，控制「用户余额不足」告警 |
| `feishu_webhook_notify_account` | `true` / `false`，同时控制「账号额度超限」与「账号被限流」两类告警 |
| `feishu_webhook_notify_ops` | `true` / `false`，控制「监控」模块告警策略是否推送飞书（默认 `false`） |
| `feishu_webhook_at_all` | `true` / `false`，推送卡片时是否 @所有人（默认 `false`） |
| `feishu_webhook_at_user_ids` | 字符串，@指定成员的飞书 open_id 列表（换行/逗号/分号分隔） |
