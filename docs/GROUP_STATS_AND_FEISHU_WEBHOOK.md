## 分组消耗额度展示

### 功能说明

系统在以下位置展示分组维度的消耗汇总（5h / 7d 实际扣费）：

| 位置 | 展示范围 | 展示内容 |
|------|---------|---------|
| **用户看板** (`/dashboard`) | 当前用户名下 API 密钥绑定的分组 | 独立的「分组消耗额度」卡片，列出各分组 5h / 7d 消耗 |
| **管理员看板** (`/admin/dashboard`) | 系统中**所有**在近 7 天内有消耗的分组 | 同样的卡片，全局视角 |
| **API 密钥管理页** | 单个绑定了分组的密钥 | 密钥下方显示该分组 5h / 7d 消耗行 |

> **说明**：展示的是分组维度的全组汇总用量，即该分组下所有用户、所有密钥的消耗总和，不限于当前用户自身。管理员看板额外展示了无关联密钥的分组。

### 界面示意

**API 密钥管理页**（分组列）：

```
[anthropic-default · Standard · ×1.0]   ← GroupBadge
5h: $0.1234  7d: $2.3456               ← 新增消耗行
```

**分组消耗额度卡片**（用户看板 / 管理员看板均相同）：

```
┌─────────────────────────────────────────────┐
│ 分组消耗额度                                 │
│ ┌──────────────────┐  ┌──────────────────┐  │
│ │ anthropic-default│  │ openai-default   │  │
│ │ 5h: $0.12        │  │ 5h: $0.00        │  │
│ │ 7d: $2.34        │  │ 7d: $0.56        │  │
│ └──────────────────┘  └──────────────────┘  │
└─────────────────────────────────────────────┘
```

卡片按 7d 实际消耗降序排列；若近 7 天无消耗记录则不渲染卡片。

### 接口说明

#### 用户侧（`/dashboard`）

```
GET /api/v1/groups/:id/stats
```

**鉴权**：需要登录 Token，且当前用户必须持有至少一个绑定该分组的 API 密钥。

**响应示例**：

```json
{
  "group_id": 1,
  "requests_5h": 42,
  "actual_cost_5h": 0.1234,
  "requests_7d": 360,
  "actual_cost_7d": 2.3456
}
```

#### 管理员侧（`/admin/dashboard`）

```
GET /api/v1/admin/dashboard/groups/windowed
```

**鉴权**：需要管理员 Token。

**响应示例**：

```json
{
  "groups": [
    {
      "group_id": 1,
      "group_name": "anthropic-default",
      "requests_5h": 42,
      "actual_cost_5h": 0.1234,
      "requests_7d": 360,
      "actual_cost_7d": 2.3456
    }
  ]
}
```

| 字段 | 说明 |
|------|------|
| `requests_5h` | 过去 5 小时的请求次数 |
| `actual_cost_5h` | 过去 5 小时的实际扣费金额（USD） |
| `requests_7d` | 过去 7 天的请求次数 |
| `actual_cost_7d` | 过去 7 天的实际扣费金额（USD） |

> `actual_cost` 对应账号侧的实际扣费，与用户侧的计费乘数无关。

### 前端组件

分组消耗卡片抽为公共组件 `frontend/src/components/dashboard/GroupStatsCard.vue`，用户看板与管理员看板均复用此组件，Props 为 `items: GroupStatItem[]`。

## 飞书 Webhook 告警

### 功能说明

在原有邮件告警的基础上，支持通过飞书机器人 Webhook 推送告警消息。每类告警内置冷却机制，防止同一告警在短时间内重复推送（默认 30 分钟内只推送一次）。

### 支持的告警类型

| 告警类型 | 触发条件 |
|---------|---------|
| **用户余额不足** | 用户余额低于设定阈值时触发（与余额低通知联动） |
| **账号额度超限** | 账号日/周/总额度触达告警阈值时触发（与账号额度通知联动） |

### 配置步骤

#### 1. 创建飞书机器人 Webhook

1. 打开飞书，进入目标群聊
2. 点击右上角「设置」→「机器人」→「添加机器人」→「自定义机器人」
3. 填写机器人名称（如 `Sub2API 告警`），完成创建
4. 复制生成的 Webhook URL，格式如：
   ```
   https://open.feishu.cn/open-apis/bot/v2/hook/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
   ```

#### 2. 在 Admin 设置页配置

进入管理员后台 → **系统设置** → **飞书 Webhook** Tab：

| 配置项 | 说明 | 默认值 |
|-------|------|--------|
| 启用飞书 Webhook | 全局开关 | 关闭 |
| Webhook URL | 飞书机器人的 Webhook 地址 | — |
| 冷却时间（分钟） | 同类告警触发后，冷却期内不重复推送 | 30 |
| 推送：用户余额不足 | 是否推送余额不足告警 | 开启 |
| 推送：账号额度超限 | 是否推送账号额度超限告警 | 开启 |

> **注意**：余额不足告警需要先在「余额低通知」中启用全局开关；账号额度超限需要在「账号配额通知」中启用并配置阈值。飞书 Webhook 配置现已从「邮件设置」Tab 移至独立的「飞书 Webhook」Tab。

### 消息格式

推送消息采用飞书文本格式，示例如下：

**余额不足告警**：
```
[Sub2API] 用户余额不足
用户：alice（alice@example.com）
当前余额：$0.0123
告警阈值：$1.0000
```

**账号额度超限**：
```
[Sub2API] 账号额度告警（日限额 / Daily）
账号：my-anthropic-account
平台：anthropic
维度：日限额 / Daily
当前用量：$9.8765 / 告警阈值：$9.5000
```

### 冷却机制说明

- 冷却以「告警类型 + 实体 ID」为维度独立计算
  - 余额告警：每个用户独立冷却
  - 账号额度告警：每个账号独立冷却
- 冷却基于 Redis，重启服务不会重置冷却状态（TTL 保存在 Redis 中）
- 冷却时间可在设置页调整，最小 1 分钟，最大 1440 分钟（24 小时）

### 与邮件告警的关系

飞书告警与邮件告警**并行触发**，互不影响：

| | 邮件告警 | 飞书 Webhook |
|-|---------|-------------|
| 余额不足 | 发送给用户本人及其额外邮箱 | 推送到飞书群 |
| 账号额度超限 | 发送给管理员通知邮箱 | 推送到飞书群 |
| 控制开关 | 各自独立 | 各自独立 |

## 相关配置键（仅供参考）

飞书 Webhook 设置存储在 `settings` 表中：

| Key | 说明 |
|-----|------|
| `feishu_webhook_enabled` | `true` / `false` |
| `feishu_webhook_url` | Webhook URL 字符串 |
| `feishu_webhook_cooldown_minutes` | 整数，单位分钟 |
| `feishu_webhook_notify_balance` | `true` / `false` |
| `feishu_webhook_notify_account` | `true` / `false` |
